Architecture

Two-plane design

Control plane: Redis with per-worker-type task queues, plus thread-based message history for persistent project context across delegations.

Data plane: Shared /workspace volume between master and all workers. Workers clone repos, write code, and produce artifacts here.

Redis Data Model

Five groups of keys, each with a clear purpose and TTL policy.

1. Task Queues per worker type

KeyTypePurposeTTL
tasks:queue:claudeLISTPending tasks for Claude workernone
tasks:queue:copilotLISTPending tasks for Copilot workernone
tasks:queue:opencodeLISTPending tasks for OpenCode workernone
tasks:processing:claudeLISTIn-flight tasks (BLMOVE target, crash recovery)none
tasks:processing:copilotLISTIn-flight tasksnone
tasks:processing:opencodeLISTIn-flight tasksnone

Task payload (JSON in list):

{
  "task_id": "uuid-abc",
  "thread_id": "thread-123",
  "instruction": "Add OAuth2 support to the existing design"
}

2. Thread Message History

thread:{thread_id}:messages   LIST — ordered history of all interactions
TTL: 7 days, refreshed on each new message

Each entry:

{
  "role": "master | claude | copilot | opencode",
  "content": "The actual design text, code diff, or instruction",
  "timestamp": "2026-05-09T10:00:00Z",
  "metadata": { "task_id": "uuid-abc", "tokens": 1200 }
}

Workers pull context with LRANGE thread:{id}:messages -5 -1 before executing. Inactive threads auto-cleanup via TTL.

3. Thread Current State

thread:{thread_id}:current_state   HASH
FieldValues / Description
statusawaiting_review | refining | implementing | complete
last_designFull text of the latest design version
last_updated_byWorker type that last updated
last_task_idTask ID of the last update
gh_pr_numberGitHub PR number (if code was pushed)
gh_repoGitHub repo (e.g. "owner/repo")
updated_atISO timestamp

A snapshot updated after each task — avoids parsing the full message history to find the current state.

4. Active Task Registry

active_tasks   HASH
Field: task_id → {"status":"running", "worker":"claude", "thread_id":"...", "started_at":"..."}

The master uses this to see what's in flight. Workers register on dequeue, remove on completion.

5. Per-Task Result Keys

KeyTypeValues
task:{id}:statusSTRINGpending | running | done | failed | cancelled
task:{id}:workerSTRINGclaude | copilot | opencode
task:{id}:thread_idSTRINGParent thread
task:{id}:descriptionSTRINGOriginal instruction
task:{id}:resultSTRINGstdout/stderr from agent CLI
task:{id}:created_atSTRINGISO timestamp
task:{id}:completed_atSTRINGISO timestamp
task:{id}:exit_codeSTRINGExit code (0 = success)
task:{id}:cancelSTRINGCancellation flag

TTL: 24 hours for all per-task keys. Thread history is the long-term record.

Master: task.py CLI

The master runs claude interactively. It calls task.py for all task and thread management. The script lives at scripts/task.py and is copied into the master image at /usr/local/bin/task.py.

Task Commands

CommandDescription
task.py enqueue --worker X --thread ID --instruction "..." LPUSH a task onto tasks:queue:X. Appends instruction to thread:{id}:messages. Refuses if thread:{id}:lock exists. Prints {"task_id": "..."}.
task.py status --id ID Prints JSON: {task_id, worker, thread_id, status, exit_code, timestamps}.
task.py result --id ID [--tail N] Prints the result field.
task.py list [--worker X] [--status X] [--thread X] [--limit N] Scans active_tasks and task:* keys with SCAN, prints summary table.
task.py wait --id ID [--timeout 300] Blocks until done/failed. Polls every 2s. On completion, deletes thread:{id}:lock.
task.py requeue-stale [--worker X] [--older-than 600] Scans tasks:processing:X and re-queues orphaned tasks.
task.py cancel --id ID Sets task:{id}:cancel = "1". Checked by worker before subprocess start.
requeue-stale edge case: If a worker crashes after writing the result but before lrem(PROCESSING, ...), the task has correct status and won't be re-queued, but a stale entry remains in the processing list. Harmless but accumulates over time.

Thread Commands

CommandDescription
task.py thread-create --id ID [--repo owner/repo]Initializes thread:{id}:current_state with status=initiated.
task.py thread-history --id ID [--tail N]Prints last N messages from thread:{id}:messages.
task.py thread-state --id IDPrints current_state hash as JSON.
task.py thread-update --id ID --status S [--design "..."] [--pr N]Updates fields in current_state. Used by master after reviewing results.
task.py thread-listLists all threads with status summary.
task.py thread-cleanup --id IDDeletes /workspace/{id}/. Prevents volume bloat.
task.py unlock --thread IDClears stale lock left by crashed master.

Example Flow

# Start a new project thread
task.py thread-create --id "add-oauth2" --repo "owner/repo"

# Delegate design to Claude worker
DESIGN_TASK=$(task.py enqueue --worker claude --thread add-oauth2 \
    --instruction "Design OAuth2 support." | jq -r '.task_id')
task.py wait --id "$DESIGN_TASK"

# Delegate review to Copilot worker
REVIEW_TASK=$(task.py enqueue --worker copilot --thread add-oauth2 \
    --instruction "Review the OAuth2 design. Find security gaps." | jq -r '.task_id')
task.py wait --id "$REVIEW_TASK"

# Delegate implementation to OpenCode worker
task.py enqueue --worker opencode --thread add-oauth2 \
    --instruction "Implement OAuth2 based on design and review."

# Check progress
task.py thread-state --id add-oauth2
task.py thread-history --id add-oauth2 --tail 10

Worker: worker.py Poll Loop

A single generic worker.py shared by all three worker types. Parameterized by environment variables:

Build Prompt Flow

Worker Behavior

  • seq Each worker runs one task at a time, sequentially.
  • ctx Before executing, fetches last HISTORY_WINDOW messages + current_state snapshot. Gives the agent full context without the master repeating it.
  • append After executing, appends its result to thread history. The next worker in the pipeline sees it.
  • sigterm On SIGTERM: lets the current subprocess finish, then exits. In practice Docker Compose sends SIGKILL after the stop grace period (default 10s), so the container is killed and the in-flight task is left in tasks:processing:* for requeue-stale to recover.
  • cap Result content capped at 10k chars when appended to thread history. Full result still in task:{id}:result.
  • lock Thread serialization: SETNX thread:{id}:lock prevents concurrent tasks on the same thread. Lock released on wait completion. Stale locks cleared by unlock.
  • cancel Task cancellation: Master sets task:{id}:cancel. Worker checks before subprocess start. Mid-execution cancellation not supported in v1.
  • health Healthcheck: 3 failures (90s) → Docker restarts container. Task recovered by requeue-stale.

Design Rationale

Python vs Go for the Orchestration Scripts

Python wins for this use case. task.py and worker.py are thin orchestration glue — CLI arg parsing, Redis commands, subprocess management. redis-py, argparse, and subprocess are mature and concise for this.

Go would add build complexity (multi-arch binaries), a Go toolchain dependency in the build pipeline, and more code for the same logic — without benefit, since the worker is single-task-at-a-time and subprocess.run() matches the model perfectly. No goroutines needed.

Shared /workspace Volume Tradeoffs

The shared volume is the data plane — where workers clone repos, write code, and produce artifacts.

Without it, you lose:

  • The master can't inspect files a worker produced (only sees text output in Redis)
  • Thread workspace isolation (/workspace/{thread_id}/)
  • A restarted worker loses in-progress clones and builds

What's not lost: all task delegation, results, and thread history flow through Redis. If workers push code to GitHub (branches/PRs), the repo is the real artifact store and the workspace is scratch space.

Verdict: Keep the shared volume. It's lightweight and gives the master a direct window into worker output. For a future iteration it could become optional, with workers using local ephemeral storage and relying on git as the sole artifact store.

Agent Session Save/Load

Claude Code supports --resume <session-id> and --continue (resumes the last session from ~/.claude/projects/). Copilot (copilot -p) and OpenCode (opencode run) are one-shot — no built-in session persistence.

This design intentionally does not rely on agent-native sessions. Instead it rebuilds context from Redis thread history before each task. This is tool-agnostic and works uniformly across all three workers.

Tradeoff: The agent's internal reasoning chain from the previous invocation is lost; each run is a cold start with conversation history injected. For v2, --resume could be added as an optimization for the Claude worker without changing the architecture.
